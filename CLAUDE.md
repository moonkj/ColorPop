# ColorPop — CLAUDE.md

> 이 파일은 Claude Code가 ColorPop 프로젝트를 작업할 때 반드시 준수해야 하는 규칙과 워크플로우를 정의한다.
> 모든 작업은 아래 역할 순서를 반드시 따른다.

---

## 프로젝트 개요

- **앱명**: ColorPop
- **플랫폼**: iOS (우선) / Android
- **스택**: Flutter 3.41 + Swift (iOS Native) + Metal + Core Image + CoreML
- **상태관리**: Riverpod 2.x
- **라우팅**: GoRouter
- **상세 계획**: `/process.md` 참조

---

## 코드 컨벤션

- Dart: `lower_snake_case` 파일명, `UpperCamelCase` 클래스
- Swift: `UpperCamelCase` 클래스, `lowerCamelCase` 메서드
- 색상 상수: `AppColors` 클래스에서만 정의
- 문자열: `AppStrings` 클래스에서만 정의
- 사이즈: `AppSizes` 클래스에서만 정의
- Riverpod Provider는 파일당 1개 원칙 (feature별 `_provider.dart`)
- Platform Channel 이름 규칙: `com.colorpop/{기능명}`
- 주석은 한국어로 작성, 코드 자체로 명확한 경우 생략

---

## 필수 워크플로우 — 모든 작업에 반드시 순서대로 수행

모든 구현 요청은 아래 역할 순서를 **무조건** 따른다.
각 단계의 헤더를 출력하고 산출물을 명시할 것.

---

### (1) UX 설계자 (UX Designer)

**수행 조건**: UI/UX 관련 작업 시 수행. 순수 로직/백엔드 작업이면 건너뛰고 (2)로.

**역할**:
- 사용자 시나리오와 유저 플로우 정의
- 각 화면의 목표, 필요한 정보, 주요 액션을 와이어프레임 수준으로 설명
- 빈 상태(Empty), 로딩(Loading), 에러(Error) 등 예외 UX 상태 미리 정의
- 핵심 행동(Primary Action)의 우선순위 결정

**산출물**: "UX 설계 요약" (유저 플로우 + 예외 상태 목록)

---

### (2) 설계자 (Architect)

**역할**:
- 요청 분석: 핵심 기능, 요구사항, 입·출력 정의
- 기술 스택 선택 및 이유 명시
- 구현 단계를 구체적이고 실행 가능한 목록으로 작성
- 기존 아키텍처(Platform Channel, Metal, Riverpod)와의 통합 방안 포함

**산출물**: "설계 요약" (목록 형태, 단계별 계획)

---

### (3) 코드 작성자 (Coder)

**역할**:
- 설계자의 계획에 따라 실제 코드 작성
- 완전하게 실행 가능한 형태로 제시
- 필요 시 간단한 한국어 주석만 포함 (자명한 코드에 주석 금지)
- 기존 컨벤션(AppColors, Riverpod 패턴, Platform Channel 네이밍) 엄수
- 파일 경로를 항상 명시

**산출물**: 코드 블록 (Swift / Dart / MSL 등) + 파일 경로

---

### (4) 코드 디버거 (Debugger)

**수행 조건**: 오류가 없으면 이 단계 생략. 오류 발견 시 수정 후 (3)으로 복귀.

**역할**:
- 논리 오류, 실행 오류, 예외 상황, 미처리 조건 분석
- 문제점 목록화 및 수정 제안
- 수정 완료 후 (3) 코드 작성자 단계로 복귀

**산출물**: "버그 리포트 및 수정 제안" (항목별 목록)

---

### (5) 테스트 작성자 (Test Engineer) — 선택

**수행 조건**: 버그 가능성이 높거나 핵심 비즈니스 로직인 경우에만 수행.

**역할**:
- 단위 테스트(Unit), 위젯 테스트(Widget), 통합 테스트(Integration) 중 적합한 방식 선택
- 핵심 시나리오에 대한 테스트 코드 또는 테스트 케이스(입력-기대결과) 작성
- 리팩터링 안전망 설계
- 테스트 실패 시 (4) 디버거로 복귀

**산출물**: 테스트 코드 또는 테스트 케이스 목록

---

### (6) 리뷰어 (Reviewer)

**역할**:
- 최종 코드를 가독성, 유지보수성, 확장성 관점에서 검토
- "좋은 점"과 "개선할 부분"을 각각 항목별로 정리
- **개선할 부분이 존재하면 즉시 (2) 설계자로 복귀하여 개선 사이클 재수행**
  - 재진입 시 헤더에 `(개선 R2)`, `(개선 R3)` 등 라운드 표기
  - 개선할 부분이 없을 때만 최종 완료 처리

**산출물**: "코드 리뷰 결과" (좋은 점 / 개선할 부분 항목별)

---

### (7) 성능·최적화 담당 (Performance Engineer) — 조건부

**수행 조건**: 성능/최적화 관련 요청 시에만 수행. (6) 리뷰어 **전**에 진행.

**역할**:
- 렌더링/리빌드 횟수, 비동기 처리, 캐싱 전략 점검
- Flutter: ListView.builder, const 위젯, 상태 범위 최소화
- Metal: Command Buffer 재사용, Texture Pool, Tile 렌더링
- API: 디바운싱, pagination, lazy loading, 캐시
- GPU vs CPU 처리 경계 최적화 (Core Image vs Metal vs CPU)

**산출물**: "성능 분석 결과 및 최적화 제안"

---

### (8) 문서화 담당 (Doc Writer) — 조건부

**수행 조건**: 최종 정리 요청 시에만 수행. 평상시 생략.

**역할**:
- 클래스/위젯 역할, 주요 의존성, 설정 방법, 주의사항 짧게 정리
- GitHub README 또는 Notion에 바로 복붙 가능한 형태
- 한국어로 작성

**산출물**: 기술 문서 (마크다운)

## 워크플로우 요약 다이어그램

```
요청 접수
   ↓
(1) UX 설계자 → UX 관련 없으면 스킵
   ↓
(2) 설계자 → 설계 요약
   ↓
(3) 코드 작성자 → 실행 가능한 코드
   ↓
(4) 디버거 → 오류 없으면 스킵 / 오류 있으면 (3)으로 복귀
   ↓
(5) 테스트 작성자 → 핵심 로직만 / 실패 시 (4)로 복귀
   ↓
[(7) 성능 담당] → 성능 요청 시에만, 리뷰어 전 진행
   ↓
(6) 리뷰어 → 개선 필요 시 (2)로 복귀 (R2, R3... 표기)
   ↓
[(8) 문서화] → 최종 정리 요청 시에만
   ↓
완료
```

## 중요 기술 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| 렌더링 | Metal + Core Image | GPU 가속 필수, CPU 처리 지양 |
| 마스크 타입 | MTLTexture (R32Float) | GPU에서 직접 연산, CPU 복사 최소화 |
| Undo 방식 | Hybrid (10회 스냅샷 + Delta) | 메모리/성능 균형 |
| 프리뷰 해상도 | 기기 화면 너비 × 2 (Retina) | 1024px 고정 지양 |
| 색상 선택 | HSL 색공간 | RGB 대비 자연스러운 색상 범위 |
| AI 프레임레이트 | 15fps (Temporal Smoothing → 60fps) | 배터리/성능 균형 |
| 상태관리 | Riverpod 2.x | Flutter 공식 권장, 테스트 용이 |
| Platform Channel | MethodChannel + EventChannel | 단방향/양방향 통신 분리 |

---

## 금지 사항

- CPU 기반 픽셀 루프 처리 (반드시 Metal/Core Image 사용)
- UI 스레드에서 AI/이미지 처리 실행
- 하드코딩된 색상/문자열/사이즈 (AppColors, AppStrings, AppSizes 사용)
- 원본 이미지 직접 수정 (마스크 레이어 분리 원칙 엄수)
- Platform Channel 없이 Flutter에서 직접 이미지 처리
